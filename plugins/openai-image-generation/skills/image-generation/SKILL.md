# 图片生成

通过 `generate_image` 创建新图片。保持用户意图，不擅自添加人物、品牌、文案或叙事情节。

## 工作流

1. 判断请求是否为创建新图。若用户要求修改、抠图或保持现有图片中的内容，说明本插件暂不支持图片编辑，不要冒充编辑工具。
2. 判断是一组相同提示词的变体，还是多个内容不同的资产：
   - 相同提示词的变体可在一次调用中使用 `n`。
   - 不同主体、用途或文案的资产分别调用，每次使用独立 prompt；不要用 `n` 代替不同提示词。
3. 整理主体、场景、风格、构图、光线、色彩、材质、文字原文和必须保持/避免的约束。
4. 信息足够时直接调用工具。仅在缺失信息会实质改变结果且无法合理推断时追问。
5. 成功后简短说明生成数量；不要在正文输出数据 URL、服务端 URL 或 API key。

## Prompt 组织

按需使用以下顺序写成简洁、完整的视觉规格，不必机械填满所有字段：

```text
用途：图片将用于哪里
主体与场景：主要对象、环境及关键细节
风格与媒介：照片、插画、3D 等
构图：画幅、视角、主体位置、需要的留白
光线与色彩：光照、氛围、配色
文字："必须逐字呈现的原文"
约束：必须保留、禁止出现、水印等
```

- 用户描述已经具体时，只做结构化整理，不额外发挥。
- 用户描述较泛时，只补充能明确提升结果的构图、光线或成品用途信息。
- 图片包含文字时，逐字引用原文并说明位置；不要改写、翻译或虚构文案。
- 对容易漂移的要求，在 prompt 末尾再次明确“只生成什么、不要出现什么”。

## 参数

- 默认只传 `prompt`，让插件使用已配置模型和服务默认值。
- 仅在用户明确提出，或目标服务能力已知时传 `size`、`quality`、`background`、`output_format` 等参数。
- `model` 只用于用户明确要求的模型覆盖；不要猜测模型名。
- `n` 只表示同一 prompt 的变体数量。
- 透明背景只在用户明确要求且服务支持时使用 `background: transparent`；本插件不承诺本地抠图或原生透明能力。

## 约束

- 不要在聊天中索要、回显或猜测 API key。
- `generate_image` 不可用时，引导用户前往“能力 -> 插件 -> 图片生成”安装插件并填写服务地址与 API key。
- 不要承诺已经检查构图、文字准确性或透明通道；当前生成结果不会自动回传给模型进行视觉检查。
- 服务拒绝某个可选参数时，移除该参数后最多重试一次；不要静默更换模型或扩大生成数量。
